// 01 — vibe coding & emails

Vibe coding is great. Except for emails.

Modern builders use ChatGPT, Claude, Cursor, Lovable, Bolt, or v0. You can ship a full product in a weekend — UI, auth, database, landing page, the works. You launch, users sign up and then they get an email.
And honestly, it's terrible...

Default template
Wrong font, wrong colours
Broken spacing
Unstyled links
Dark mode chaos
Landed in spam
// why this happens

Because email HTML is different from web HTML. And AI doesn't know the constraints unless you explicitly tell it. So you get unbranded emails that break in Gmail, lose styles in Outlook, and go straight to spam.

where it breaks
// 02 — where AI breaks emails

AI generates web HTML. Email needs something different.

AI usually generates normal HTML. Email needs AI to generate email-safe HTML.

✗ Tailwind classes
✗ External CSS
✗ Flexbox layouts
✗ Custom fonts
✗ Modern HTML tags
✗ Dark mode styles
✗ Responsive layouts
✗ Inline style conflicts

The output looks fine in preview. Then it breaks in Gmail, Outlook strips the styles, and users see something completely different from what you built. This is why most vibe-coded apps ship default templates because the fix isn't obvious.

// 03 — why default auth emails look bad

Default templates work. They just look terrible.

Tools like Supabase give you working email templates. But they're generic, minimal, and unbranded>

// the typical workflow

Copy template → edit in dashboard → save → break layout → user receives it.
No versioning. No preview. No rollback. No safe deploy. So people stop touching them.

The default stays. And every email your app sends looks like it came from a demo project, not a real product.

fixing it
// 04 — the safe way to handle emails

Systemize your emails.

Use a structured file layout: one layout, simple HTML, inline styles, no runtime CSS, no JS. Small templates that are predictable, testable, and safe to change.

emails/
  layout.html   ← one layout
  confirm.html  ← reuse the structure
  invite.html

This structure makes emails easy to maintain and helps AI to work much better with your emails.

// 05 — how to use AI safely for emails

Give constraints. Get good output.

AI works well for emails but only when you're specific. A good prompt makes all the difference.

✓ Good prompt Generate an email-safe HTML template.
Use tables for layout.
Use inline styles only.
No flexbox. No external CSS.
Must work in Gmail and Outlook.

✗ Bad prompt — "Make a nice email template"

Best workflow: use AI to generate a base template once, fix the layout, then reuse the structure.

previewing & shipping
// 06 — always preview before sending

Never deploy emails blind.

1 Preview in browser to catch obvious layout breaks early
2 Send a test email to a real inbox, not a preview pane
3 Check Gmail + Outlook, every time
4 Then deploy with version history so you can roll back
// the bad workflow

Edit → save → user receives → panic. Most tools don't give you preview, version history, or rollback. So most builders skip all three and find out when users complain.

// 07 — deliverability

Good-looking emails that never arrive.

You test that emails work locally. You send to yourself and it's all good. You ship... and users never get the email. This is where many vibe-coded apps fail silently.

Email delivery depends on domain settings, not just HTML. These don't throw errors, it shows up as emails in spam, delayed, blocked, or simply never sent.

✗ SPF record missing
✗ Wrong sender address
✗ DKIM not enabled
✗ Using default SMTP
✗ Domain not verified
✗ Bad domain reputation

The minimum safe setup:

✓ Custom domain
✓ Verified sender address
✓ SPF record configured
✓ DKIM enabled
✓ Domain verified in your email provider (Resend / Postmark / SMTP)
✓ Tested in Gmail + Outlook with a real user account

Most builders do this last. It should be done first. Set up your domain, verify it, configure SPF + DKIM, send a test, check the spam folder. Only then ship emails.
